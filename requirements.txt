==============================================================
ADAPTIVE INTERVIEW DESIGN — AI Build-Plan Generator (FP2)
==============================================================

PURPOSE
-------
This document specifies the adaptive question flow the system uses to
interview a user about their software idea. The goal is NOT a fixed
questionnaire — it is a conversation that reacts to each answer, asks
sharper follow-ups when answers are vague, and skips what has already
been covered. The collected answers are later turned into a tailored
sequence of build prompts.

The interview must cover eight areas before it is considered complete.
Each area below lists: the GOAL of the question, an OPENING QUERY, a set
of ADAPTIVE FOLLOW-UPS that depend on the user's answer, and a COVERAGE
CHECK that tells the system when it has enough to move on.


HOW THE ADAPTIVITY WORKS
------------------------
1. Ask one question at a time, in plain language.
2. Before each new question, read everything the user has already said.
   - If an earlier answer already covered an upcoming area, skip the
     opening query and confirm instead ("Earlier you mentioned X — is
     that still the plan?").
3. If an answer is vague, short, or generic, ask a follow-up that makes
   it concrete before moving on. Do not advance on a non-answer.
4. Mirror the user's vocabulary. If they are non-technical, avoid jargon;
   if they are technical, allow it.
5. Track which of the eight areas are "covered" vs "open." End the
   interview only when all eight are covered.
6. Keep a running understanding of the project and let later questions
   build on earlier answers ("Since this is for warehouse staff on
   tablets, should it work offline?").


==============================================================
THE EIGHT COVERAGE AREAS
==============================================================

(1) PAIN POINTS — what specific problem are you solving?
--------------------------------------------------------
GOAL: A clear, specific problem statement, not a vague wish.
OPENING QUERY:
  "In one or two sentences, what problem are you trying to solve?"
ADAPTIVE FOLLOW-UPS:
  - If the answer is broad ("make things easier"):
      "Can you give me a recent, concrete example of when this was a
       problem?"
  - If they describe a solution instead of a problem ("I want an app
    that does X"):
      "What goes wrong today that makes you want that app?"
  - If multiple problems are named:
      "Which of these is the most important to solve first?"
COVERAGE CHECK: You can restate the core problem in one sentence and the
  user agrees with it.


(2) DATA SOURCES — what data are you consuming?
-----------------------------------------------
GOAL: Identify what information the software works with.
OPENING QUERY:
  "What information or data does this software need to work with?"
ADAPTIVE FOLLOW-UPS:
  - If they name a source ("our sales spreadsheet"):
      "Roughly how big is it, and how often does it change?"
  - If they say "I'm not sure":
      "What would the software need to KNOW to be useful? Think about
       the facts it reads or stores."
  - If several sources are named:
      "Do these sources need to be combined, or used separately?"
COVERAGE CHECK: You have a list of the main data the system reads or
  stores, with a rough sense of size and update frequency.


(3) DATA ACCESS — how are you getting the data?
-----------------------------------------------
GOAL: Understand how the data physically arrives in the system.
OPENING QUERY:
  "How does that data get into the system — typed in, uploaded,
   pulled from another service?"
ADAPTIVE FOLLOW-UPS:
  - If "from another system/API":
      "Do you know the name of that system, and do you have access
       (login, API key) to it?"
  - If "uploaded files":
      "What format are they — Excel, CSV, PDF, images?"
  - If "entered by hand":
      "Who enters it, and how often?"
COVERAGE CHECK: For each major data source, you know the access method
  (manual entry, file upload, API/integration, or database).


(4) END RESULT — what are you producing?
----------------------------------------
GOAL: Define the concrete output the user gets.
OPENING QUERY:
  "When the software works perfectly, what does it give you?"
ADAPTIVE FOLLOW-UPS:
  - If vague ("useful insights"):
      "What would you actually look at or download — a report, a
       dashboard, an alert, a file?"
  - If they describe a process, not an output:
      "And at the end of that, what is the thing you walk away with?"
  - If multiple outputs:
      "Which output is the one that matters most?"
COVERAGE CHECK: You can name the primary output and its form (e.g.,
  on-screen dashboard, downloadable PDF, email alert).


(5) STAKEHOLDERS & CONSUMERS — who will own and use this?
---------------------------------------------------------
GOAL: Identify who runs it and who uses its output.
OPENING QUERY:
  "Who is going to USE this day to day, and who OWNS it?"
ADAPTIVE FOLLOW-UPS:
  - If only one role is given:
      "Is the person who uses it the same person who maintains it?"
  - If "customers / the public":
      "Roughly how many people, and do they need accounts/logins?"
  - If internal team:
      "How many people, and do they share one view or have different
       roles?"
COVERAGE CHECK: You know the primary user(s), the owner, and rough scale
  (one person, a team, or the public).


(6) AUDIENCE TYPE — business owners or developers?
--------------------------------------------------
GOAL: Calibrate how technical the users are (affects UI and wording).
OPENING QUERY:
  "Are the people using this more like business/everyday users, or
   technical/developer users?"
ADAPTIVE FOLLOW-UPS:
  - If "business / non-technical":
      "Should it feel like a simple website with buttons and forms?"
  - If "developers":
      "Would they expect something like an API, a CLI, or raw data
       access rather than a polished screen?"
  - If "a mix":
      "Who is the primary audience if you had to choose one?"
COVERAGE CHECK: You can classify the audience as non-technical,
  technical, or mixed-with-a-primary.


(7) CURRENT PROCESS — how is this solved today?
-----------------------------------------------
GOAL: Understand the existing workaround so the build can replace it.
OPENING QUERY:
  "How do you handle this today, before the software exists?"
ADAPTIVE FOLLOW-UPS:
  - If "we don't / manually":
      "Walk me through the manual steps you do now."
  - If a tool is named ("we use Excel"):
      "What part of that current way is the most painful or slow?"
  - If "nothing exists yet":
      "Is there a similar product you wish you could just use?"
COVERAGE CHECK: You understand the current workflow and the specific
  step the new software is meant to improve or replace.


(8) INTERACTION MODEL — chat conversation or batch report?
----------------------------------------------------------
GOAL: Decide how the user interacts with the software.
OPENING QUERY:
  "Do you picture talking to it back-and-forth, or setting it up once
   and getting results automatically?"
ADAPTIVE FOLLOW-UPS:
  - If "back-and-forth / chat":
      "Should it remember past conversations, or start fresh each time?"
  - If "automatic / scheduled":
      "How often should it run — on demand, daily, or when new data
       arrives?"
  - If unsure:
      "When you imagine using it, are you clicking a button and waiting,
       or typing questions and reading answers?"
COVERAGE CHECK: You can describe the interaction as conversational,
  on-demand/batch, or scheduled/automatic.


==============================================================
ENDING THE INTERVIEW
==============================================================
- Before finishing, briefly summarize the project back to the user in a
  few sentences and ask: "Did I get that right, or is anything missing?"
- Let the user correct or add to any area.
- Only mark the interview complete once all eight areas are covered and
  the user confirms the summary.
- The confirmed answers become the input for generating the adaptive
  prompt sequence.
